# IT-da 구현 백로그

이 문서는 2026-04-27 기준 현재 리포지토리 구현 상태를 바탕으로 남은 작업을 정리한 실행 백로그다.
초기 설계 단계의 "처음부터 무엇을 만들까"보다, 지금 시점에서 "무엇이 이미 들어갔고 다음에 무엇을 완성할까"에 초점을 둔다.

## 1. 현재 기준 요약

| 영역 | 상태 | 메모 |
| --- | --- | --- |
| 회원/인증 | 완료 | 세션 기반 로그인, 회원가입, 기본 권한 흐름 구현 |
| 이력서/프로필 | 완료 | `Resume`, `ResumeSkill`, `ResumeAttachment`, `ProfileImage`, 경력/스킬 수정 흐름 구현 |
| 제안서 작성 | 완료 | `WRITING` 저장, `MATCHING` 등록, 포지션/요구 스킬 편집, 수정용 edit draft 복제 정책 구현 |
| AI 브리프 | 완료 | 제안서 자유 입력 기반 초안 생성 및 매핑 구현 |
| AI 인터뷰 | 완료 | 대화 메시지 저장, 제안서 초안 점진 갱신, 인터뷰 드래프트 동기화 구현 |
| 추천 엔진 MVP | 구현 | 실행 생성/재사용, 추가 추천, 상태 폴링, 결과 저장, 추천 이유 생성, 이력서 임베딩 캐시 구현 |
| 매칭 MVP | 구현 | 요청, 수락/거절, 계약 시작 확인, 취소 요청/철회/확인, 후기, 완료 확인 구현 |
| 운영 시드 | 구현 | `local`/`demo` 프로필에서 프리랜서 50명, 시드 제안서 2건, 추천 결과/매칭 샘플 적재 |
| 운영자 기능 | 미구현 | 운영자 전용 화면과 추천 품질 관리 도구는 아직 없음 |
| 파일 확장 도메인 | 보류 | `proposal_attachments`, 계약서/완료 증빙 모델은 아직 없음 |

## 2. 이미 반영된 기반

### 2.1 프리랜서 프로필과 추천 입력

- `Resume.status`, `Resume.writingStatus`, `publiclyVisible`, `aiMatchingEnabled`가 모두 코드에 반영돼 있다.
- 스킬/경력 수정 시 `ResumeEmbeddingService`를 통해 임베딩을 갱신한다.
- 프로필 이미지와 이력서 첨부는 `StoredFile` 메타데이터를 통해 관리한다.

### 2.2 제안서와 AI 입력 흐름

- `Proposal`, `ProposalPosition`, `ProposalPositionSkill` 저장 구조가 구현돼 있다.
- 제안서 총 예산은 포지션별 `headCount * unitBudget` 합산으로 계산한다.
- `MATCHING` 상태 제안서는 조건부로 원본 재사용 또는 새 `WRITING` 복제본에서 수정한다.
- AI 브리프는 `proposal.raw_input_text`를 입력으로 사용한다.
- AI 인터뷰는 `proposal_ai_interview_message`에 대화 이력을 저장하고, 누적 대화를 `raw_input_text`와 제안서 초안에 다시 반영한다.

### 2.3 추천 엔진 MVP

- `RecommendationRun`은 동일 fingerprint면 재사용하고, 추가 추천은 `excludedResumeIds`를 포함한 새 fingerprint로 생성한다.
- 추천 실행은 `PENDING -> RUNNING -> COMPUTED/FAILED` 상태로 관리한다.
- 스케줄러가 3초마다 대기 중 실행을 가져와 처리한다.
- 추천 결과는 `RecommendationResult`에 저장하고, 추천 이유 생성 실패는 전체 실행 실패로 간주하지 않는다.
- 이력서 임베딩은 `ResumeEmbedding`에 캐시하며 source hash가 바뀌면 갱신한다.

### 2.4 매칭 MVP

- 추천 결과에서 매칭 요청을 생성할 수 있다.
- 프리랜서는 요청 수락/거절을 처리할 수 있다.
- `ACCEPTED -> IN_PROGRESS`는 양측 계약 시작 확인 시점으로 관리한다.
- `ACCEPTED` 취소 요청은 상대방 확인 또는 24시간 경과 시 자동 취소된다.
- `IN_PROGRESS -> COMPLETED`는 양측 후기 작성과 완료 확인이 모두 끝났을 때만 허용한다.
- 매칭 상태 변화에 따라 `proposal_position.status`를 `OPEN/FULL/CLOSED`로 재계산한다.

## 3. 남은 우선순위

### 3.1 프리랜서 직접 지원 흐름

현재 매칭 생성은 추천 결과 기반 요청 흐름에 맞춰져 있다.
추천 없이 프리랜서가 프로젝트에 직접 지원하는 엔트리와 상태 전이를 별도 UX로 붙이는 작업이 남아 있다.

필요 산출물:

- 직접 지원 생성 엔드포인트와 화면 진입점
- 클라이언트 요청과 프리랜서 지원을 구분하는 UI 문구
- 동일 `proposal_position + resume` 활성 매칭 중복 방지 검증 유지

검증 포인트:

- 추천 경로와 직접 지원 경로가 같은 중복 제약을 사용하는지
- `MATCHING`이 아니거나 `OPEN`이 아닌 포지션에는 직접 지원이 차단되는지

### 3.2 추천 품질 고도화와 운영 가시성

현재 추천 MVP는 동작하지만, 품질을 설명하고 운영하기 위한 관측 정보는 아직 얕다.

필요 산출물:

- 후보군 수, 필터 탈락 사유, 점수 분해를 더 잘 보여주는 운영 뷰
- 추천 실패/빈 결과 원인 파악용 로그와 화면
- fingerprint 재사용 정책과 캐시 무효화 전략 문서화

검증 포인트:

- 빈 결과, 실패, 추가 추천, 재실행 케이스를 각각 재현할 수 있는지
- 점수 설명과 최종 순위가 일관되게 보이는지

### 3.3 운영자 도구

운영자 전용 화면은 아직 없다.
직무/스킬 정규화와 추천 품질 검토를 위한 최소 도구가 필요하다.

필요 산출물:

- canonical `position` 관리 화면
- `Skill` 검색/정리 화면
- 추천 실행 이력과 실패 케이스 조회 화면

검증 포인트:

- 일반 사용자와 운영자 접근 제어가 분리되는지
- 운영자 화면이 실제 추천 저장 모델을 읽는지

### 3.4 파일 기반 확장 도메인

현재 제안서 파일 경로는 존재하지만 정식 도메인 연관은 없다.
계약서/완료 증빙도 텍스트/시각 기반 상태로만 처리한다.

필요 산출물:

- `proposal_attachments` 도입 여부 결정
- 계약서/완료 증빙 파일 모델 필요성 검토
- `StoredFile`과 확장 자산 엔티티 책임 경계 정리

검증 포인트:

- 파일 업로드 추가 후 기존 프로필/이력서 업로드 흐름과 충돌하지 않는지
- 파일 유무가 상태 전이 조건을 지나치게 복잡하게 만들지 않는지

## 4. 현재 운영 규칙

- 모든 활성 회원은 클라이언트 기능을 사용할 수 있다.
- 프리랜서 참여는 `resume.status = ACTIVE`를 전제로 한다.
- 검색/목록 노출은 `writing_status = DONE`과 `publicly_visible = true`를 전제로 한다.
- AI 추천 노출은 여기에 `ai_matching_enabled = true`를 추가로 본다.
- `proposal.total_budget_*`는 입력값이 아니라 포지션 예산 합산 결과다.
- 추천 실행은 `proposal.status = MATCHING`이고 `proposal_position.status = OPEN`일 때만 허용한다.
- 연락처는 `matching.status = ACCEPTED` 이후에만 공개한다.
- `PROPOSED`는 정원을 점유하지 않고, `ACCEPTED`와 `IN_PROGRESS`만 정원을 점유한다.
- `proposal_attachments`, 파일 기반 계약서, 완료 증빙은 아직 정식 도메인에 포함하지 않는다.

## 5. 다음 작업 순서

1. 프리랜서 직접 지원 엔트리와 UI를 추가한다.
2. 추천 실행/결과/실패 케이스를 운영 관점에서 볼 수 있는 최소 화면을 만든다.
3. 추천 품질 로그와 점수 설명을 정리해 디버깅 비용을 낮춘다.
4. 운영자용 직무/스킬 관리 도구를 붙인다.
5. 파일 기반 확장 도메인 도입 여부를 결정하고 필요 시 별도 백로그로 분리한다.

## 6. 병렬화 가능한 작업

- 추천 품질 가시성 보강과 운영자 화면 설계
- 직접 지원 UX 작업과 상태 전이 테스트 보강
- 파일 확장 도메인 설계와 현재 업로드 책임 분석

아래 작업은 병렬화하면 충돌 가능성이 크다.

- `Matching` 상태 규칙 변경과 `MatchingQueryService` 뷰모델 재설계
- 추천 fingerprint 정책 변경과 추천 실행 재사용 규칙 수정

## 7. 검증 우선순위

1. 상태 전이 테스트: `WRITING -> MATCHING`, `PROPOSED -> ACCEPTED -> IN_PROGRESS -> COMPLETED`, 취소 분기
2. 정원 재계산 테스트: `FULL -> OPEN` 복귀와 `CLOSED` 차단
3. 추천 실행 테스트: 재사용, 추가 추천, 빈 결과, 실패
4. 권한 테스트: 소유자/프리랜서/비당사자 접근 제어
5. 시드 시나리오 테스트: `local` 프로필 실행 시 대시보드, 추천, 매칭 흐름 수동 검증

## 8. 완료 기준

현재 백로그 기준으로는 아래가 만족되면 MVP 이후 단계로 넘어갈 수 있다.

- 추천 외 경로의 직접 지원이 들어와도 현재 매칭 규칙과 충돌하지 않는다.
- 추천 결과와 매칭 상태를 운영자가 추적할 수 있다.
- 추천 품질 이슈를 로그/화면으로 재현하고 설명할 수 있다.
- 파일 확장 도메인 도입 여부가 결정돼 설계가 분기되지 않는다.
