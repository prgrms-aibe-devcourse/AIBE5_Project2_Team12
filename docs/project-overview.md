# IT-da 프로젝트 문서

이 문서는 `IT-da`의 제품 기획과 현재 구현 상태를 한 번에 파악하기 위한 기준 문서다.
기획 내용은 팀 논의 메모와 유저 플로우 정리본을 기준으로 작성했고, 구현 상태는 현재 리포지토리 코드를 기준으로 정리했다.

## 1. 프로젝트 개요

- 프로젝트명: `IT-da`
- 팀명: `Generic<4>`
- 서비스 정의: IT 업종 한정 프로젝트 의뢰인과 IT 프리랜서를 연결하는 AI 기반 매칭 플랫폼
- 핵심 가치: 자유 텍스트 기반 프로젝트 설명을 AI 브리프로 구조화하고, 설명 가능한 추천으로 후보 비교와 선택을 돕는 것

### 핵심 사용자

- 클라이언트: 프로젝트 요구사항을 빠르게 정리하고 적합한 프리랜서를 찾고 싶은 사용자
- 프리랜서: 자신의 스킬, 경력, 선호 업무를 구조적으로 노출해 적합한 프로젝트에 추천받고 싶은 사용자
- 운영자: 데이터 품질과 추천 품질을 관리하는 내부 사용자

같은 회원이 클라이언트와 프리랜서 역할을 동시에 수행할 수 있다.
기본적으로 모든 활성 회원은 클라이언트 기능을 사용할 수 있고, 이력서를 등록한 회원은 프리랜서 기능도 사용할 수 있다.
프리랜서 추천 노출은 `Resume.publiclyVisible`, AI 추천 후보 포함은 `Resume.aiMatchingEnabled`로 별도 제어한다.

## 2. 문제 정의

기존 프리랜서 매칭 흐름은 프로젝트 요구사항을 구조적으로 정리하기 어렵고, 후보 탐색과 비교에 시간이 많이 든다.
IT-da는 이 문제를 아래 두 축으로 풀고자 한다.

- 입력 보조: 클라이언트의 자유 입력을 AI 브리프로 구조화해 제안서 초안 작성 부담을 낮춘다.
- 비교 보조: 단순 리스트업보다 추천 근거, 스킬 일치도, 비교 가능한 후보 정보를 제공해 선택 마찰을 줄인다.

## 3. 핵심 서비스 흐름

서비스의 기본 흐름은 아래와 같다.

1. 프리랜서가 프로필을 등록한다.
2. 클라이언트가 단일 텍스트로 프로젝트를 입력한다.
3. AI 브리프가 요구사항, 기술 스택, 역할, 경력 수준, 예산 범위를 구조화한다.
4. 클라이언트가 제안서를 검토하고 수정한 뒤 제출한다.
5. AI 추천 엔진이 후보를 선별하고 Top 3를 추천한다.
6. 클라이언트가 추천 후보와 지원자 리스트를 비교해 요청을 보낸다.
7. 프리랜서가 수락하면 매칭이 성립하고 연락처가 공개된다.
8. 매칭 성립 후 참여 관계는 `논의중 -> 시작 -> 종료` 상태로 관리된다.

## 4. 사용자별 플로우

### 4.1 클라이언트 플로우

1. 회원가입 및 로그인
2. 새 프로젝트 생성
3. 자유 텍스트 입력
4. AI 브리프 생성
5. 구조화된 제안서 검토 및 수정
6. 저장 또는 제출
7. AI 추천 결과 확인
8. 추천 후보와 지원자 비교
9. 매칭 요청 전송
10. 수락 후 참여 진행

핵심 포인트는 아래와 같다.

- 제안서 작성 UX는 "직접 작성"보다 "AI가 만든 초안을 수정"하는 흐름에 가깝다.
- AI 브리프 실패 시 수동 폼 fallback이 반드시 필요하다.
- 제안서는 임시 저장 상태와 제출 완료 상태를 구분한다.
- 여러 프로젝트를 동시에 관리할 수 있다.

### 4.2 프리랜서 플로우

1. 회원가입 및 로그인
2. 프로필 작성
3. 스킬, 경력, 자격, 선호 업무, 포트폴리오 등록
4. 추천 노출 허용 여부 설정
5. 프로젝트 탐색 또는 추천 결과 확인
6. 프로젝트 지원
7. 매칭 요청 수락 또는 거절
8. 참여 상태 관리

핵심 포인트는 아래와 같다.

- 추천 노출은 프리랜서가 직접 켜고 끌 수 있어야 한다.
- `aiMatchingEnabled`는 AI 추천 후보군 포함 여부만 제어하고, 직접 지원 자체를 막지는 않는다.
- 프리랜서 기능은 별도 역할 컬럼이 아니라 이력서 등록 여부로 활성화한다.
- 다중 프로젝트 참여가 가능하다.
- 참여 상태는 클라이언트가 의사결정할 때 참고할 수 있어야 한다.

### 4.3 운영자 플로우

- 추천 품질 모니터링
- 스킬 정규화와 데이터 정합성 관리
- 시드 데이터와 기준 데이터 관리
- 캐시 상태와 추천 결과 품질 확인

## 5. AI 기능 설계

### 5.1 AI 브리프

클라이언트의 단일 텍스트 입력을 구조화된 제안서 폼으로 변환한다.

예상 출력 필드는 아래와 같다.

- 요구사항
- 기술 스택
- 역할 또는 직무
- 경력 수준
- 예상 예산 범위
- 유사 성공 사례

실패 시에는 구조화 결과 대신 수동 입력 폼으로 전환한다.

### 5.2 추천 엔진

추천 엔진은 하이브리드 방식으로 설계한다.

1. 하드 필터
2. 유사도 계산
3. LLM 기반 설명 생성

세부 원칙은 아래와 같다.

- 하드 필터는 RDBMS 기반 조건 필터링으로 후보군을 먼저 줄인다.
- 구조화된 필드가 충분하면 코사인 유사도 또는 벡터 유사도 방식을 우선 활용한다.
- 구조화가 부족하면 룰 베이스 fallback으로 품질 저하를 완충한다.
- LLM 설명은 응답으로 노출되는 Top 3 후보에 대해서만 생성한다.
- 매 요청마다 전부 재생성하지 않고 세션 또는 캐시를 활용해 비용을 제어한다.

### 5.3 설명 가능한 추천

추천 결과 UI는 블랙박스를 최소화하는 방향을 목표로 한다.

- 스킬 일치도 시각화
- AI 코멘트
- AI 추천 면접 질문

## 6. 도메인 모델 요약

현재 기획 기준의 핵심 도메인은 아래와 같다.

- 회원: 클라이언트, 프리랜서, 운영자
- 이력서 또는 프로필: 프리랜서의 스킬, 경력, 선호 업무, 포트폴리오
- 제안서: 클라이언트가 작성하는 프로젝트 문서
- 제안서 포지션: 하나의 제안서 안에서 실제 모집 단위가 되는 포지션
- 직무 마스터: 재사용 가능한 공용 직무 기준값
- 매칭: 요청, 수락, 확정 흐름을 표현하는 관계
- 파일: 이력서, 제안서, 프로필 이미지 등 업로드 자산

현재 논의 흐름상 개념 구분은 아래가 적절하다.

- `position`: 직무 마스터 테이블
- `proposal_position`: 실제 모집 단위
- `matching`: `proposal_position` 기준으로 성립하는 요청과 수락 흐름
- MVP에서는 별도 `project` 테이블을 두지 않고, 매칭 이후의 참여 이력을 `matching`에서 관리한다.
- 서비스 역할은 별도 컬럼으로 저장하지 않고, `Resume` 존재 여부와 공개 설정으로 파생한다.
- `proposal.total_budget_*`는 전체 프로젝트 예산이고, `proposal_position.unit_budget_*`는 1인 기준 예산이다.

## 7. 상태 모델

### 7.1 제안서 상태

- `WRITING`: 작성 중, 임시 저장 가능 상태
- `DONE`: 마지막으로 제출된 추천 기준이 유효한 상태

`DONE` 상태의 제안서를 수정할 때는 아래처럼 나눈다.

- 추천 입력에 영향이 있는 변경: `WRITING`으로 되돌린 뒤 재제출
- 첨부파일, 표시 순서 같은 비핵심 변경: `DONE` 유지 가능

### 7.2 참여 상태

- `논의중`: 매칭 성립 직후 협의 단계
- `시작`: 양측 동의 후 진행 중
- `종료`: 양측 동의 후 완료

### 7.3 매칭 상태

- 요청
- 수락
- 확정
- 거절

## 8. 핵심 운영 규칙

- 연락처는 매칭 이전에 공개하지 않는다.
- 매칭 이후에만 상호 연락처를 확인할 수 있다.
- 프리랜서와 클라이언트 모두 다중 프로젝트가 가능하다.
- 추천 결과가 모두 거절되면 제안서 보완을 유도한다.
- 제안서의 추천 입력이 바뀌면 추천과 설명을 재계산한다.
- 개인정보는 저장하되 화면에서는 마스킹 노출을 우선 고려한다.

## 9. MVP 범위

### MVP 필수

- AI 브리프 기반 제안서 구조화
- 프리랜서 프로필 및 이력서 관리
- 추천 엔진
- 설명 가능한 추천 결과
- 후보 비교 및 선택
- 요청 -> 수락 -> 확정의 매칭 흐름
- 매칭 이후 참여 상태 관리

### 중간 우선순위

- 검색 기능
- 이력서 게시판 또는 유사 탐색 기능

### MVP 제외

- SNS 기능
- 결제 시스템
- 관심 프리랜서 알림
- 신규 프리랜서 등록 시 자동 재랭킹 알림
- 비 IT 업종 확장
- 외부 데이터 대량 연동

## 10. 기술 방향

기획 기준 기술 스택 방향은 아래와 같다.

- 백엔드: Spring Boot, Spring Data JPA, Querydsl
- 데이터베이스: PostgreSQL, H2
- 보안: Spring Security
- 프론트엔드: Thymeleaf, Bootstrap, Bootswatch, jQuery 검토
- 인프라: Docker Compose, 싱글 서버 배포
- 파일 저장: 로컬 파일 저장 + `ResourceHandler` 노출
- AI: 외부 LLM API

아키텍처는 일정상 모놀리식 애플리케이션을 기본으로 하고, 추후 확장 시 AI 추천 영역의 분리를 검토한다.

## 11. 현재 구현 상태

현재 리포지토리 기준으로 이미 구현된 항목은 제한적이며, 아래 범위가 확인된다.

### 구현됨

- 회원가입 및 로그인 화면
- 세션 기반 폼 로그인
- 회원 도메인
- 이력서 도메인
- 스킬 도메인
- 파일 업로드 및 로컬 저장
- 프로필 이미지, 제안서, 이력서 파일 경로 분리

### 아직 구현되지 않음

- 이력서 기반 프리랜서 진입 제어 플로우
- 프로젝트 제안서 작성 UI
- AI 브리프 생성
- 추천 엔진
- 지원자 비교 및 선택
- 매칭 요청, 수락, 확정 플로우
- 매칭 이후 참여 상태 관리
- 운영자용 운영 화면

### 코드 기준 확인된 구현 포인트

- 인증은 Spring Security의 폼 로그인 기반이며 세션을 사용한다.
- 파일은 로컬 경로에 저장하고 정적 리소스 경로로 노출한다.
- 이력서는 회원과 1:1로 연결된다.
- 경력 데이터는 JSON payload로 저장된다.
- 스킬은 별도 엔티티로 관리된다.

## 12. 구현과 기획 사이의 간극

현재 코드베이스는 제품 전체보다 계정, 이력서, 스킬, 파일 저장의 기초 도메인에 집중되어 있다.
기획의 핵심인 AI 브리프, 제안서, 추천, 매칭은 아직 제품 문서 수준에서만 정의된 상태다.

따라서 다음 단계는 아래 순서가 적절하다.

1. 제안서와 제안서 포지션의 도메인 규칙 확정
2. 이력서 기반 프리랜서 진입 조건과 화면 흐름 확정
3. 추천 입력 필드 구조화
4. 추천 엔진의 최소 동작 버전 정의
5. AI 브리프의 요청/응답 스키마 정의
